# 15_CALLS_AUTOMANUAL_BINARY_RUBRIC.md

Дата фиксации: 23.06.2026  
Проект: `onboarding_cross_node`  
Назначение: бинарный мануал оценки звонков новичка по автомануалу. Мануал нужен, чтобы AI-модель, тимлид и аналитик одинаково отвечали на вопросы `да/нет` и считали процент по каждому этапу продаж.

---

## 1. Главный принцип

Оценка звонка строится по бинарным вопросам:

```text
да / нет / неприменимо / конфликт
```

Каждый вопрос имеет вес. Сумма весов внутри этапа равна `1.00` / `100%`.

Итог по этапу:

```text
stage_score = сумма весов вопросов с answer = yes / сумма весов применимых вопросов
```

Пример:

| Вопрос | Вес | Ответ |
|---|---:|---|
| Поздоровался | 0.10 | yes |
| Представился | 0.15 | yes |
| Сообщил, что из Банка Точка | 0.25 | yes |
| Уточнил удобство разговора | 0.15 | no |
| Сообщил причину звонка | 0.35 | no |

```text
stage_score = 0.10 + 0.15 + 0.25 = 0.50 = 50%
```

---

## 2. Источники оценки

| Источник | Как использовать |
|---|---|
| транскрибация звонка | основной источник фактов |
| базовый мануал | источник финальных вопросов и комментариев для модели |
| сегментный мануал | дополнительный слой оценки под сегмент клиента/новичка |
| словарь продуктов Точки | проверка продуктов, сокращений, описаний, статуса продажи |
| карточка клиента / роль | сегмент, СНО, род деятельности, история взаимодействий, цель, возражения |
| экспертная разметка | если в комментарии к вопросу указано, что работу вопроса нужно смотреть на экспертной разметке |

---

## 3. Статусы ответа

| Статус | Когда ставим | Участвует в расчёте |
|---|---|---|
| `yes` | действие/условие явно выполнено | да |
| `no` | действие должно было быть выполнено, но не выполнено | да |
| `not_applicable` | вопрос неприменим к этому звонку | нет |
| `conflict` | данные противоречат друг другу | нет, нужно ручное уточнение |
| `not_enough_data` | транскрибация неполная или не даёт проверить вопрос | нет, нужна проверка |

Для звонков правило жёстче, чем для собеседования:

```text
Если действие обязательно для этапа и его нет в транскрибации, ставим no.
```

---

## 4. Общие веса этапов звонка

| Этап | Вес в итоговой оценке |
|---|---:|
| Установление контакта | 0.15 |
| Выявление потребности | 0.30 |
| Презентация продукта | 0.25 |
| Работа с возражениями | 0.15 |
| Тест на сделку / завершение звонка | 0.15 |

Формула:

```text
call_score =
  contact_score * 0.15 +
  needs_score * 0.30 +
  presentation_score * 0.25 +
  objections_score * 0.15 +
  close_score * 0.15
```

Если этап `Работа с возражениями` неприменим, потому что возражений не было, его вес перераспределяется пропорционально между применимыми этапами или помечается как `not_applicable` — в зависимости от логики отчёта.

Рекомендуемый вариант для аналитики:

```text
не снижать итог за отсутствие возражений, если клиент не возражал.
```

## 5. Обязательная логика обращения к другим таблицам

Некоторые вопросы нельзя оценивать только по транскрибации. Если в комментарии к вопросу указано использовать другой источник, модель обязана это сделать.

### 5.1. Словарь продуктов Точки

Использовать для вопросов:

- презентовал ли оператор продукт банка;
- спросил ли мнение клиента о продукте;
- подкрепил ли выгоду расчётами по продукту;
- связал ли продукт с проблемой клиента;
- обработал ли возражение по релевантности продукта;
- предложил ли действие для покупки/подключения продукта.

Ожидаемые поля словаря:

| Поле | Как использовать |
|---|---|
| `Продукт Точки` | основное название продукта |
| `Сегмент` | для сегментной релевантности |
| `Тип продукта` | классификация продукта |
| `Описание продукта` / `Контекст` | проверка, что продукт подходит под боль клиента |
| `Сокращения` | поиск продукта в транскрибации по неформальным названиям |
| `Статус` | брать только действующие/продаваемые продукты |

Правило:

```text
Продукт засчитывается только если он найден в словаре продуктов Точки и имеет статус “продаётся”, “действующий” или аналогичный активный статус.
```

Если продукт не найден:

```text
answer = no
comment = "Продукт не найден в словаре продуктов Точки или не имеет активного продаваемого статуса."
```

Если продукт найден по сокращению:

```text
answer может быть yes, но в evidence нужно указать нормализованное название продукта.
```

---

### 5.2. Карточка клиента / роль

Использовать для вопросов:

- каким бизнесом занимается клиент;
- релевантна ли проблема клиента;
- соответствует ли продукт сегменту и ситуации клиента;
- корректно ли отработаны ожидаемые возражения;
- достигнуто ли целевое действие звонка.

Ожидаемые поля карточки/роли:

| Поле | Как использовать |
|---|---|
| `Род деятельности` | определение сегмента и бизнес-контекста |
| `СНО` | контекст для бухгалтерии, отчётности, налогов, расчётов |
| `Взаимодействия ранее` | учитывать историю контакта и уже озвученные продукты |
| `Информация о клиенте` | контекст для гипотезы и цели звонка |
| `Целевое действие` | проверка завершения звонка |
| `Возражения` | проверка работы с ожидаемыми сопротивлениями |
| `Тон` | контекст поведения клиента, не оценка оператора сам по себе |

---

### 5.3. Сегментный мануал

Использовать, если известен сегмент клиента/новичка:

- грузоперевозки;
- стройка;
- селлеры;
- розница;
- core / mix;
- ВЭД;
- другой сегмент, если он есть в таблице.

Правило:

```text
Базовый мануал оценивается всегда.
Сегментный мануал добавляется вторым слоем, если сегмент определён.
```

Сегмент нельзя угадывать только по одному слову. Нужно опираться на карточку клиента, роль, род деятельности или прямое содержание диалога.

---

### 5.4. Экспертная разметка

Если в инструкции к вопросу написано:

```text
Работу вопроса нужно посмотреть на экспертной разметке.
```

то модель должна:

1. не придумывать критерий сама;
2. использовать экспертную разметку как источник правила;
3. если экспертная разметка недоступна — вернуть `not_enough_data` и комментарий, что требуется источник.

## 6. Этап 1 — Установление контакта

Сумма весов этапа = `1.00`.

| ID | Бинарный вопрос | Вес | Инструкция для модели |
|---|---|---:|---|
| `contact_01` | Уточнил ли оператор, удобно ли клиенту разговаривать? | 0.15 | Засчитывать `yes`, если оператор спросил про удобство разговора и дал клиенту возможность ответить |
| `contact_02` | Поздоровался ли оператор? | 0.10 | Засчитывать любое корректное приветствие |
| `contact_03` | Представился ли оператор, назвав своё имя? | 0.15 | Засчитывать, если оператор назвал своё имя; только “это банк” без имени = `no` |
| `contact_04` | Сообщил ли оператор, что звонит из Банка Точка? | 0.25 | Засчитывать, если клиенту понятно, что звонок от Банка Точка / Точки |
| `contact_05` | Сообщил ли оператор причину звонка? | 0.35 | Причина должна объяснять, зачем оператор звонит именно сейчас / по какому поводу / с какой пользой для клиента |

Комментарий по весам:

```text
Приветствие и имя — базовая гигиена.
Банк и причина звонка важнее, потому что дают клиенту контекст и снижают сопротивление.
```

### Аналитика этапа

Если `contact_score = 1.00`:

```text
Установление контакта выполнено полностью: оператор поздоровался, представился, обозначил Банк Точка, уточнил удобство и сообщил причину звонка.
```

Если меньше `1.00`, комментарий должен перечислить, какие обязательные элементы не выполнены.

## 7. Этап 2 — Выявление потребности

Сумма весов этапа = `1.00`.

| ID | Бинарный вопрос | Вес | Инструкция для модели |
|---|---|---:|---|
| `needs_01` | Спросил ли оператор, каким бизнесом занимается клиент? | 0.20 | Засчитывать, если оператор выяснил сферу, формат работы, клиентов, процессы или подтвердил уже известный бизнес-контекст |
| `needs_02` | Спросил ли оператор клиента, с какими проблемами и трудностями он сталкивается в бизнесе? | 0.30 | Под проблемами считать сложности, препятствия, неудобства, неэффективность, ручной труд, риски, кассовые разрывы, задержки, потери времени |
| `needs_03` | Уточнил ли оператор, как часто клиент сталкивается с проблемами и трудностями в бизнесе? | 0.15 | Засчитывать, если есть уточнение регулярности: разово, часто, каждый месяц, периодически, постоянно |
| `needs_04` | Уточнил ли оператор у клиента, как он решает проблемы и трудности в бизнесе? | 0.20 | Засчитывать, если оператор выяснил текущий способ решения, альтернативу, ручной процесс или текущего поставщика/банк |
| `needs_05` | Уточнил ли оператор у клиента, сколько ресурсов он тратит на решение проблем и трудностей в бизнесе? | 0.15 | Ресурсы = деньги, время, сотрудники, нервы, операционные действия, потери, комиссии, просрочки |

Важное правило:

```text
Если оператор не произносит слово “проблема”, но выясняет сложность/боль/неудобство клиента по смыслу, вопрос может быть yes.
```

Если оператор просто спрашивает “вам актуально?” без погружения в бизнес, это не считается полноценным выявлением.

## 8. Этап 3 — Презентация продукта

Сумма весов этапа = `1.00`.

| ID | Бинарный вопрос | Вес | Инструкция для модели |
|---|---|---:|---|
| `presentation_01` | Презентовал ли оператор какие-либо продукты банка? | 0.20 | Использовать словарь продуктов Точки. Засчитывать только действующие/продаваемые продукты |
| `presentation_02` | Спрашивал ли оператор мнение клиента хотя бы об одном представленном продукте банка? | 0.20 | Использовать словарь продуктов Точки. Засчитывать, если оператор проверил реакцию клиента: актуально, интересно, что думаете, подходит ли |
| `presentation_03` | Подкрепил ли оператор выгоду от презентованных продуктов банка конкретными расчётами? | 0.25 | Засчитывать расчёт, сравнение, экономию, стоимость, срок, цифру, пример с деньгами. Пример: “разница состава около 300 тысяч” |
| `presentation_04` | Презентовал ли оператор продукты банка, сославшись на озвученную клиентом проблему? | 0.35 | Сначала найти озвученную проблему клиента, потом проверить, что продукт связан с этой проблемой. Если нет выявленной проблемы, вопрос = `no` |

Обязательное обращение к словарю продуктов:

```text
Для всех вопросов этапа “Презентация продукта” модель должна сверять продукт со словарём продуктов Точки.
```

Если продукта нет в словаре или статус неактивный:

```text
presentation_01 = no
presentation_02 = no, если мнение спрашивали только по невалидному продукту
presentation_03 = no, если расчёт был только по невалидному продукту
presentation_04 = no, если связка была только с невалидным продуктом
```

## 9. Этап 4 — Работа с возражениями

Этап применим только если клиент реально выразил возражение, сомнение, отказ, перенос, неактуальность, недоверие, сравнение с конкурентом или ограничение.

Если возражений не было:

```text
objections_stage_status = not_applicable
```

Сумма весов этапа = `1.00`.

| ID | Бинарный вопрос | Вес | Инструкция для модели |
|---|---|---:|---|
| `objections_01` | Пытался ли оператор снять возражения клиента по представленным продуктам банка? | 0.30 | Имеет смысл только если возражения были. Засчитывать, если оператор уточнил причину и ответил по сути, а не просто продолжил презентацию |
| `objections_02` | Уточнил ли оператор, остались ли у клиента сомнения после ответа на его возражения по представленным продуктам банка? | 0.20 | Засчитывать сам факт вопроса о снятии сомнения, а не результат согласия клиента |
| `objections_03` | Приводил ли оператор примеры или сравнения, иллюстрирующие использование или выгоду представленных продуктов банка? | 0.15 | Засчитывать любое сравнение, кейс, метафору, опыт клиента, пример из жизни или работы банка. Не обязательно напрямую по продукту, если пример помогает убеждению |
| `objections_04` | Сравнивал ли оператор предложенные продукты или условия по ним с предложениями конкурентов? | 0.15 | Под конкурентами понимать другие банки, платёжные системы, онлайн-сервисы и любые организации, предлагающие аналогичные продукты/услуги |
| `objections_05` | Высказывал ли клиент возражения по поводу релевантности использования представленных продуктов банка? | 0.20 | Фильтрующий вопрос. Должен идти первым в логике анализа блока релевантности. Под релевантностью понимать возражения по цене, необходимости, возможности использования, целесообразности применения для бизнеса клиента |

Важное правило по `objections_05`:

```text
Это вопрос не про качество оператора, а про наличие типа возражения у клиента.
Если клиент такого возражения не высказывал, ставим not_applicable для последующих под-вопросов, завязанных на релевантность.
```

Если клиент сказал “дорого”, “не нужно”, “не пользуюсь”, “нам это не подходит”, “не вижу смысла”, “уже есть в другом банке” — это возражение по релевантности/целесообразности.

## 10. Этап 5 — Тест на сделку и завершение звонка

Сумма весов этапа = `1.00`.

| ID | Бинарный вопрос | Вес | Инструкция для модели |
|---|---|---:|---|
| `close_01` | Предложил ли оператор клиенту совершить действия, необходимые для приобретения продукта? | 0.30 | Засчитывать действия: подать заявку, оформить продукт, подключить продукт, перейти к оплате, подтвердить согласие |
| `close_02` | Договорился ли оператор с клиентом о конкретном следующем шаге с указанием даты или срока? | 0.30 | Следующий шаг = повторный звонок, отправка коммерческого предложения, подключение продукта в срок, встреча, заявка. Нужна конкретика по сроку/действию |
| `close_03` | Предлагал ли оператор клиенту списать денежные средства с его счёта? | 0.20 | Фильтрующий вопрос. Имеет смысл только если продукт/сценарий допускает списание денег со счёта |
| `close_04` | Согласился ли клиент на списание денежных средств с его счёта? | 0.20 | Имеет смысл только при `close_03 = yes`. Засчитывать любые формы подтверждения: “да”, “угу”, “ага”, “согласен”, “давайте” и подобные. Формальная юридическая формулировка не требуется |

Если сценарий продукта не предполагает списание:

```text
close_03 = not_applicable
close_04 = not_applicable
```

Если оператор должен был предложить списание по сценарию продукта, но не предложил:

```text
close_03 = no
close_04 = not_applicable
```

## 11. Критичные ошибки и ограничения итоговой оценки

Ограничители применяются после расчёта базового процента.

| Ошибка | Ограничение |
|---|---:|
| Оператор пообещал клиенту то, чего нельзя обещать | итог звонка не выше 50% |
| Оператор презентовал продукт, который не продаётся / неактивен в словаре продуктов | этап презентации = 0% по этому продукту |
| Оператор презентовал продукт без выявления потребности | итог звонка не выше 70% |
| Не было выявления потребности вообще | итог звонка не выше 65% |
| Нет завершения или следующего шага | итог звонка не выше 80% |
| Клиент согласился, но оператор не зафиксировал действие | итог звонка не выше 75% |
| Оператор давил на клиента или спорил вместо обработки возражения | итог звонка не выше 70% |
| Оператор исказил условия продукта | итог звонка не выше 60% |

Критичная ошибка должна быть подтверждена evidence из транскрибации.

## 12. Аналитика по этапу

По каждому этапу модель должна возвращать не только процент, но и краткий смысловой вывод.

Формат:

```json
{
  "stage": "Установление контакта",
  "score": 0.50,
  "score_percent": 50,
  "answers": [
    {
      "question_id": "contact_01",
      "answer": "no",
      "weight": 0.15,
      "evidence": "В начале звонка оператор сразу перешёл к причине звонка, не уточнив удобство разговора."
    }
  ],
  "what_done_well": [
    "Оператор поздоровался, представился и сообщил, что звонит из Банка Точка."
  ],
  "what_failed": [
    "Не уточнил удобство разговора.",
    "Не объяснил причину звонка достаточно ясно."
  ],
  "stage_summary": "Контакт установлен формально, но клиент не получил полного контекста звонка. Нужно начинать с удобства разговора и понятной причины контакта."
}
```

## 13. Интерпретация процентов по этапу

| Процент | Интерпретация |
|---:|---|
| 0–20% | этап провален |
| 21–40% | сильная просадка |
| 41–60% | частично выполнено, качество нестабильное |
| 61–75% | рабочий минимум |
| 76–89% | хороший уровень |

## 16. Запреты для модели

Модель не должна:

- засчитывать продукт без проверки по словарю продуктов Точки;
- засчитывать презентацию, если оператор просто перечислил продукты без связи с клиентом;
- считать “вам актуально?” полноценным выявлением потребности;
- штрафовать блок возражений, если клиент не возражал;
- ставить `yes` без цитаты/смыслового evidence из транскрибации;
- придумывать продуктовые условия, если их нет в словаре/мануале;
- игнорировать комментарии из базового мануала;
- игнорировать сегментный мануал, если сегмент известен;
- смешивать оценку звонков с оценкой собеседования.
