# Onboarding Analytics Dashboard — пакет проектной памяти

Этот пакет фиксирует договорённости по будущей системе аналитики новичков кросс-продаж.

Цель пакета — не сделать таблицу, а сохранить образ результата, архитектурную логику, зоны ответственности GPT/Codex и план внедрения, чтобы проект не потерялся между чатами, Google Sheets, сервером и ручными сборками.

## Что мы хотим получить

Не просто отчёт по одному новичку и не ещё одну Google-таблицу.

Целевой продукт:

1. **Большой дашборд по новичкам** — сравнение потоков, сегментов, статусов, рисков, hard/soft-метрик и динамики.
2. **Карточка конкретного новичка** — полный профиль с собеседованием, тестовым днём, погружением, ботами, звонками, операционкой, итоговым тестом и выводом.
3. **Сравнение новичков** — выбор нескольких людей и сравнение по одинаковым критериям.
4. **Аналитика по этапам обучения** — где проседают не только люди, но и сами курсы/тесты/боты.
5. **AI-анализ** — формирование портрета новичка, рисков, рекомендаций тренеру и итогового решения.
6. **HTML/Excel/Markdown-отчёты** — как выгрузка из системы, а не как основной источник истины.

## Главный принцип

Источник истины должен быть на сервере.

Google Sheets можно использовать как:

- импорт;
- экспорт;
- временный источник старых данных;
- ручную админку на переходном этапе.

Но не как боевое ядро.

```text
Серверная база / JSON / SQLite / Postgres
→ API
→ форма сбора карточки
→ AI-анализ
→ dashboard
→ HTML / Excel / Google Sheets export
```

## Как пользоваться этим пакетом

Файлы читать по порядку:

1. `01_PRODUCT_VISION.md` — образ результата.
2. `02_CURRENT_SOURCES.md` — какие источники данных уже найдены.
3. `03_KEYS_AND_DATA_MODEL.md` — ключи, сущности и связи.
4. `04_NEWBIE_CARD_V1.md` — структура карточки новичка.
5. `05_AI_ANALYSIS_AND_SCORING.md` — как должен анализировать AI.
6. `06_DASHBOARD_SPEC.md` — будущий дашборд и сравнения.
7. `07_GPT_VS_CODEX_WORKFLOW.md` — кто что делает.
8. `08_ROADMAP.md` — фазы внедрения.
9. `09_CODEX_PROMPTS.md` — готовые тексты задач для Codex.
10. `10_DECISIONS_AND_OPEN_QUESTIONS.md` — зафиксированные решения и вопросы.

## Важное ограничение

Codex не жечь на то, что можно сделать здесь.

GPT делает:

- анализ Google Drive;
- проектирование структур;
- Markdown/HTML/табличные шаблоны;
- промпты;
- постановки задач;
- нормализацию логики;
- подготовку ТЗ.

Codex нужен только как руки на сервере:

- чтение серверного проекта;
- правки Node/API/БД;
- webhook;
- deploy/restart/logs;
- работа через MCP/server access.


## Обновление v2: фокус на шкалах оценки и аналитике

После первых серверных фаз стало ясно: рабочее место сборки данных нужно, но оно не является конечным продуктом.

Добавлены ключевые документы:

- `11_EVALUATION_RUBRICS_BY_STAGE_V1.md` — базовая методология оценки по этапам: собеседование, тестовый день, погружение, учебные агенты, реальные звонки, операционка, выпускной тест, итоговые scores и ограничения статусов.
- `12_ANALYTICS_DASHBOARD_AND_REPORT_SPEC_V1.md` — целевая логика dashboard, read-only viewer-доступа, карточки новичка, HTML-отчётов и следующих фаз.

Новый главный принцип:

```text
Сначала фиксируем шкалы и критерии оценки.
Потом делаем ручной слой аналитики.
Потом AI.
Потом dashboard и HTML-отчёты.
```

`Completeness` — это только технический индикатор наличия данных.  
Целевой продукт должен показывать статистику, оценки, риски и рекомендации по каждому новичку.
