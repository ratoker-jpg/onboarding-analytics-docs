# 14_INTERVIEW_HARD_SOFT_BINARY_RUBRIC.md

Дата фиксации: 23.06.2026  
Проект: `onboarding_cross_node`  
Назначение: бинарный мануал оценки кандидата/новичка по транскрибации собеседования. Мануал нужен, чтобы HR, тимлид и AI-модель одинаково оценивали soft, hard и сегментное понимание.

---

## 1. Главный принцип

Оценка собеседования строится не по субъективной шкале `0–3`, а по бинарным вопросам:

```text
да / нет / не проверялось
```

Каждый вопрос имеет вес. Сумма весов внутри каждого блока равна `1.00` / `100%`.

Итог по блоку:

```text
block_score = сумма весов вопросов с answer = yes / сумма весов применимых вопросов
```

Важно: если тему не проверяли на собеседовании, нельзя ставить `no`. Для этого есть статус `not_checked`.

---

## 2. Источники оценки

Использовать только подтверждённые источники:

| Источник | Как использовать |
|---|---|
| TXT-транскрибация собеседования | основной источник фактов |
| заметки HR | дополнительный источник, если в них есть наблюдения по ответам кандидата |
| заметки тимлида | дополнительный источник по hard/sales fit |
| ролёвка на собеседовании | основной источник по технологии продаж и клиентскому подходу |
| банк вопросов HR/тимлида | источник того, какие темы реально проверялись |
| таблица критериев HR | источник soft/hard-граф и мест оценки |
| сегментные требования | источник критериев по грузоперевозкам, стройке, селлерам |
| словарь продуктов Точки | использовать, если кандидат в ролёвке или ответе привязывает боль клиента к продукту |

---

## 3. Статусы ответа

| Статус | Когда ставим | Участвует в расчёте |
|---|---|---|
| `yes` | В транскрибации/заметках есть подтверждение навыка | да |
| `no` | Тема была проверена, но навык не проявился | да |
| `not_checked` | Тему не проверяли или данных недостаточно | нет |
| `conflict` | Источники противоречат друг другу | нет, нужно ручное уточнение |

Правило:

```text
Нет данных ≠ плохой ответ.
Если нет вопроса/ситуации, где навык можно было проявить, ставим not_checked.
```

---

## 4. Формат evidence

По каждому вопросу модель обязана возвращать evidence:

```json
{
  "question_id": "soft_responsibility_01",
  "answer": "yes",
  "weight": 0.30,
  "evidence": "Кандидат описал ситуацию, где сам выбрал способ решения проблемы клиента и объяснил последствия.",
  "source": "interview_transcript",
  "comment": "Навык проявлен на конкретном примере."
}
```

Если evidence нет, ставить `yes` нельзя.

---

## 5. Итоговая структура оценки

Общий итог собеседования:

| Блок | Вес в итоговой оценке |
|---|---:|
| Soft skills | 0.50 |
| Hard / sales fit | 0.35 |
| Сегментное понимание | 0.15 |

Формула:

```text
interview_score = soft_score * 0.50 + hard_score * 0.35 + segment_score * 0.15
```

Если в блоке больше 50% веса `not_checked`, итог по блоку считается предварительным:

```text
block_confidence = low
```

### 6.2. Мотивация — 100%

Смотрим, проявляет ли кандидат желание работать именно в активных телефонных продажах.

| ID | Бинарный вопрос | Вес | Как проверять |
|---|---|---:|---|
| `soft_motivation_01` | Проявляет ли кандидат желание работать в активных телефонных продажах? | 0.35 | Ищем прямые ответы про звонки, продажи, работу с клиентами, готовность к активному контакту |
| `soft_motivation_02` | Понимает ли кандидат, что роль включает звонки, отказы и повторяющиеся действия? | 0.25 | Засчитываем, если кандидат сам говорит о рутине, отказах, необходимости держать темп |
| `soft_motivation_03` | Может ли кандидат объяснить, почему эта роль ему подходит? | 0.25 | Засчитываем не общую фразу “хочу развиваться”, а связку опыта/интереса/сильных сторон с ролью |
| `soft_motivation_04` | Есть ли у кандидата реалистичные ожидания от нагрузки и результата? | 0.15 | Засчитываем, если нет ожидания быстрых лёгких продаж без отказов и контроля |

Красные флаги:

- кандидат не хочет звонить;
- говорит, что активные продажи ему не подходят;
- хочет “просто консультировать”, без ответственности за результат;
- ожидает быстрый доход без рутины и отказов.

- избегает самостоятельных решений.

## 8. Сегментное понимание

### 8.1. Когда оценивать сегмент

Сегмент оценивается, если на собеседовании проверяли конкретный сегмент или кандидат претендует на конкретное направление:

- `Грузоперевозки`;
- `Стройка`;
- `Селлеры`;
- `Микс / core`, если нет жёсткой сегментной привязки.

Если сегмент не обсуждался, ставить `not_checked`.

---

### 8.2. Общая шкала сегмента — 100%

| ID | Бинарный вопрос | Вес | Как проверять |
|---|---|---:|---|
| `segment_understanding_01` | Понимает ли кандидат, как устроен бизнес сегмента? | 0.30 | Нужны процессы, роли, деньги, документы, цикл сделки |
| `segment_understanding_02` | Называет ли кандидат типовые боли сегмента? | 0.30 | Боли должны быть бизнесовыми, не общими “дорого/неудобно” |
| `segment_understanding_03` | Может ли кандидат связать боли сегмента с продуктами банка? | 0.25 | Использовать словарь продуктов Точки и учитывать продаваемый статус продукта |
| `segment_understanding_04` | Использует ли кандидат сегментный язык без пустого жаргона? | 0.15 | Термины должны быть уместны и объяснены по смыслу |

---

### 8.3. Сегмент: грузоперевозки

Засчитывать понимание, если кандидат корректно говорит про часть этих контекстов:

```text
ТТН, АТИ, рейсы, водители, топливо, топливные карты, оплата от юрлиц, кассовые разрывы, документы, задержки оплат, регулярные перевозки, поиск клиентов
```

Минимум для `segment_understanding_01 = yes`:

```text
кандидат объясняет не менее 2 процессов сегмента и хотя бы 1 денежную/документную боль.
```

### 8.4. Сегмент: стройка

Засчитывать понимание, если кандидат корректно говорит про часть этих контекстов:

```text
тендеры, КС-2, КС-3, закупка материалов, подрядчики, генподрядчики, СРО, закрывающие документы, долгие оплаты, кассовые разрывы, сдача объекта
```

Минимум для `segment_understanding_01 = yes`:

```text
кандидат объясняет цикл строительного проекта или документооборот/оплаты, а не просто говорит “стройка”.
```

## 9. Обращение к другим таблицам и справочникам

Некоторые вопросы нельзя оценивать только по тексту ответа кандидата.

| Что оцениваем | Куда обращаться | Зачем |
|---|---|---|
| Упоминание продукта в ролёвке | словарь продуктов Точки | проверить, что продукт существует, действующий и продаётся |
| Связка боли клиента с продуктом | словарь продуктов Точки + описание продукта | проверить, что продукт реально решает заявленную боль |
| Сегментное понимание | сегментные требования / сегментный мануал | проверить, что кандидат говорит не общими словами |
| Ролёвка с клиентом | карточка роли/клиента, если есть | учитывать сегмент, СНО, контекст, возражения, целевое действие |
| Возражения в ролёвке | карточка роли/клиента + банк вопросов | понять, было ли возражение и как его надо было отработать |

Правило по продуктам:

```text
Если кандидат называет продукт не так, как в словаре, нужно проверять сокращения и синонимы.
Если продукт найден только по сокращению, это можно засчитать.
Если продукт имеет статус “не продаётся”, “архив”, “неактивен” или аналогичный — связку с продуктом не засчитывать.
```

## 13. Запреты для модели

Модель не должна:

- ставить `yes` без evidence;
- ставить `no`, если тема не проверялась;
- засчитывать продукт без проверки по словарю продуктов, если вопрос зависит от продукта;
- засчитывать сегментное понимание по одному жаргонному слову;
- делать вывод “кандидат слабый” без указания конкретных неподтверждённых вопросов;
- смешивать оценку собеседования с оценкой реальных звонков.
