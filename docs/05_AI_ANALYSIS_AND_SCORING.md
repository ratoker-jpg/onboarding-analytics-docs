# AI-анализ и шкалы оценки

## Задача AI

AI не должен просто пересказывать данные.

Он должен:

1. Нормализовать разные источники.
2. Выделить факты.
3. Оценить hard/soft/learning/discipline/risk.
4. Сравнить этапы между собой.
5. Найти динамику.
6. Дать рекомендации тренеру.
7. Сформировать итоговый портрет новичка.

## Основные оценочные оси

### 1. Hard score

Что входит:

- знание продуктов;
- знание СНО;
- понимание условий продажи;
- корректность презентации;
- работа с продуктовой логикой;
- точность формулировок;
- отсутствие опасных обещаний;
- понимание сегмента.

Шкала: `0–100`.

### 2. Soft score

Что входит:

- контакт;
- тон;
- вежливость;
- уверенность;
- умение слушать;
- отсутствие давления;
- адаптация к клиенту;
- управление диалогом.

Шкала: `0–100`.

### 3. Learning score

Что входит:

- принимает ли обратную связь;
- меняет ли питч после ОС;
- есть ли рост между стартом/серединой/финалом;
- как проходит материалы;
- задаёт ли вопросы;
- делает ли выводы.

Шкала: `0–100`.

### 4. Discipline score

Что входит:

- прохождение маршрута;
- выполнение задач;
- заполнение целей;
- статусы;
- просрочки;
- метрики на трубке;
- стабильность.

Шкала: `0–100`.

### 5. Risk score

Что входит:

- hard-пробелы;
- soft-проблемы;
- слабая динамика;
- провалы в реальных звонках;
- ошибки в итоговом тесте;
- операционная недисциплина;
- красные флаги.

Шкала: `0–100`, где 100 = максимальный риск.

Уровни:

| Risk score | Уровень |
|---:|---|
| 0–25 | низкий |
| 26–55 | средний |
| 56–75 | высокий |
| 76–100 | критичный |

## Оценка звонков по мануалу

Оценка должна использовать два слоя:

```text
Базовый мануал + Сегментный мануал
```

### Базовый мануал

Блоки:

1. Установление контакта.
2. Выявление потребностей.
3. Презентация продукта.
4. Работа с возражениями.
5. Закрытие.

### Сегментный мануал

Блок зависит от сегмента:

- грузоперевозки;
- стройка;
- селлеры;
- микс / core;
- розница;
- ВЭД, если используется.

## Формат анализа одного звонка

AI должен вернуть структурированный JSON:

```json
{
  "call_summary": "краткий вывод по звонку",
  "result": "купил / отказ / перезвон / сброс / другое",
  "scores": {
    "contact": 0,
    "needs": 0,
    "presentation": 0,
    "objections": 0,
    "closing": 0,
    "segment_adaptation": 0,
    "overall": 0
  },
  "manual_checks": [
    {
      "question_id": "base_contact_01",
      "question": "Уточнил ли оператор, удобно ли клиенту разговаривать?",
      "answer": "yes / no / partial / not_applicable",
      "evidence": "короткая цитата или пересказ",
      "comment": "пояснение"
    }
  ],
  "client_signals": [
    "сам ведёт учёт",
    "не видит срочности"
  ],
  "strong_points": [
    "корректно уточнил текущий способ учёта"
  ],
  "risk_points": [
    "слишком рано перешёл к закрытию"
  ],
  "dangerous_promises": [
    "неточно описал оплату / создал ощущение бесплатного периода"
  ],
  "coach_comment": "что разобрать с новичком"
}
```

## Анализ учебных диалогов с ботом

Для каждого диалога учитывать:

- портрет роли;
- критерий успеха;
- критерий провала;
- целевое действие;
- ожидаемые возражения;
- тон клиента;
- транскрибацию;
- результат.

AI должен ответить:

- попал ли кандидат в роль;
- отработал ли ожидаемое сопротивление;
- выполнил ли целевое действие;
- не допустил ли критерий провала;
- какие клиентские сигналы заметил/пропустил;
- какие навыки проявились.

## Анализ динамики

Сравнивать:

```text
тестовый день → погружение → учебные боты → реальные звонки старт → середина → выпуск
```

AI должен выделять:

- рост;
- стагнацию;
- просадку;
- повторяющиеся ошибки;
- перенос обратной связи в практику;
- улучшение структуры звонка.

## Итоговый портрет новичка

Формат:

```json
{
  "candidate_profile": {
    "summary": "короткий портрет",
    "hard_score": 0,
    "soft_score": 0,
    "learning_score": 0,
    "discipline_score": 0,
    "risk_score": 0,
    "risk_level": "низкий / средний / высокий / критичный",
    "strengths": [],
    "growth_zones": [],
    "red_flags": [],
    "coach_recommendations": [],
    "final_decision": "ready / ready_with_control / needs_practice / high_risk / not_recommended"
  }
}
```

## Принцип доказательности

AI не должен писать выводы без опоры.

Для каждого важного вывода желательно иметь:

```text
факт → интерпретация → рекомендация
```

Пример:

```text
Факт: в трёх звонках подряд новичок закрывает до явного согласия клиента.
Интерпретация: риск давления и потери контакта.
Рекомендация: отдельно отработать мягкое закрытие и проверку готовности клиента.
```

## Запрещённые стили вывода

Не писать:

- “в целом молодец” без конкретики;
- слишком мягкие выводы без решения;
- абстрактные рекомендации;
- длинную воду;
- готовые реплики вместо анализа, если это не запрошено.

## Хороший стиль вывода

Писать:

- конкретно;
- по блокам;
- с фактами;
- с рисками;
- с рекомендациями;
- с итоговым решением.

# Обновление v2: обязательная привязка к рубрикам

Документ `11_EVALUATION_RUBRICS_BY_STAGE_V1.md` является главным источником методологии оценки.

AI-анализ нельзя строить как свободный пересказ. Любой AI-вывод должен опираться на:

1. этап;
2. критерии этапа;
3. шкалу 0–100;
4. объяснение оценки;
5. факты из источников;
6. риски;
7. рекомендации.

## Базовый контракт AI-оценки

AI должен возвращать структурированный JSON:

```json
{
  "stage": "interview | test_day | immersion | training_agents | calls_start | calls_middle | calls_final | ops | final_test",
  "scores": {
    "hard_score": 0,
    "soft_score": 0,
    "learning_score": 0,
    "discipline_score": 0,
    "call_quality_score": 0,
    "ops_score": 0,
    "final_test_score": 0,
    "risk_score": 0,
    "overall_score": 0
  },
  "evidence": [
    {
      "source": "interview_transcript | training_bot_dialog | call_file | manual_input | imported_snapshot",
      "fact": "конкретный факт",
      "impact": "как факт влияет на оценку"
    }
  ],
  "strengths": [],
  "growth_zones": [],
  "red_flags": [],
  "summary": "",
  "recommendation": ""
}
```

## Запрет

AI не должен:

- ставить оценку без объяснения;
- смешивать completeness и качество;
- оценивать звонок по названию продукта;
- подтягивать чужие диалоги по команде;
- игнорировать красные флаги.
