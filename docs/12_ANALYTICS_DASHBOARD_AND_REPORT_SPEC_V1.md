# ANALYTICS_DASHBOARD_AND_REPORT_SPEC_V1.md

Дата фиксации: 23.06.2026  
Проект: `onboarding_cross_node`  
Назначение: вернуть фокус проекта с “хранилища файлов” на полноценную аналитику новичков, дашборды и финальную карточку/отчёт.

---

## 1. Что важно зафиксировать

Текущая админка после Phase 2B стала рабочим местом сбора данных:

- создать/выбрать новичка;
- связать старые ключи;
- импортировать доступные источники;
- загрузить TXT, скриншоты, Teachbase-файлы;
- видеть готовность данных.

Но это только **слой сбора**.

Целевой продукт шире:

```text
Дашборд новичков
→ карточка любого новичка
→ полная статистика по этапам
→ оценки hard / soft / learning / discipline / risk
→ анализ собеседования, тестового дня, погружения, учебных агентов, звонков, операционки, выпускного теста
→ финальный HTML-отчёт по новичку, который хранится на сервере и открывается по ссылке
```

Главная ошибка текущего состояния: система показывает “есть / нет”, но почти не показывает “что это значит”.

## 2. Главные роли доступа

### Admin

Админка нужна для сборки данных.

URL:

```text
/onboarding_cross/admin-v1.html
```

Доступ:

```text
ADMIN_KEY
```

Можно:

- создавать новичков;
- добавлять source links;
- запускать импорты;
- загружать файлы;
- заполнять ручные оценки;
- формировать отчёты.

### Viewer

Dashboard просмотра нужен для тимлидов/HR/руководителей.

URL:

```text
/onboarding_cross/dashboard.html
```

Доступ:

```text
VIEWER_KEY
```

Можно:

- смотреть dashboard;
- открывать карточку новичка;
- смотреть статистику, оценки, рекомендации и отчёты.

Нельзя:

- редактировать;
- запускать импорты;
- загружать файлы;
- менять source links;
- создавать ключи.

## 4. Карточка новичка

Карточка должна быть аналитической страницей, а не складом файлов.

### Верх карточки

- ФИО;
- base_key;
- сегмент;
- направление;
- статус;
- риск;
- дата тестового дня;
- дата старта погружения;
- готовность данных;
- дата последнего анализа;
- ссылка на HTML-отчёт.

### Главный вывод

Показывать:

- общий вывод;
- рекомендацию по выпуску;
- сильные стороны;
- риски;
- что контролировать после выпуска;
- рекомендацию тимлиду.

### Scores

- Hard skills;
- Soft skills;
- Обучаемость;
- Дисциплина;
- Качество звонков;
- Операционка;
- Выпускной тест;
- Overall;
- Risk.

Каждый score должен иметь:

- число 0–100;
- уровень;
- объяснение;
- источник: ручная оценка / AI / смешанная.

### Вкладки

1. Обзор.
2. Собеседование.
3. Тестовый день.
4. Погружение.
5. Учебные агенты.
6. Звонки.
7. Операционка.
8. Выпускной тест.
9. AI-вывод.
10. История отчётов.

## 5. Как использовать текущую админку

Текущая админка — это нижний слой:

```text
сбор данных
```

Она не должна быть финальным viewer dashboard.

Нужно сохранить её для:

- импорта;
- загрузки материалов;
- ручного заполнения;
- проверки completeness.

Но финальный пользовательский просмотр должен быть отдельным:

```text
dashboard.html
```

## 7. HTML-отчёт

Сервер должен уметь генерировать HTML-файл:


Отчёт должен быть read-only и открываться по ссылке.

## 8. Важные переименования

### “Бот учебки”

В UI заменить на:

```text
Учебные агенты
```

Технически можно оставить `bot_training`.

### “Продукт” в звонках

Заменить на:

```text
Контекст / тема звонка
```

Сделать необязательным.

Оценка звонка идёт по этапам продаж и мануалу, не по продукту.

### Completeness

Не считать completeness финальной аналитикой.

Completeness показывает только:

```text
готовы ли данные для анализа
```

## 9. Этапы реализации

### Phase 3A — Analytics Model & Rubrics

- зафиксировать шкалы;
- зафиксировать scoring model;
- зафиксировать сущности;
- не кодить большой dashboard.

### Phase 3B — Manual Analytics UI

- добавить ручные оценки по этапам;
- добавить candidate_scores;
- показать read-only dashboard с ручными scores;
- viewer access.

### Phase 3C — AI Analysis

- AI-анализ собеседования;
- AI-анализ звонков;
- AI-анализ учебных агентов;
- сборка итогового профиля.

### Phase 3D — Report Snapshots

- генерация HTML-отчёта;
- история отчётов;
- ссылка на отчёт.

### Phase 3E — Full Dashboard

- фильтры;
- сравнение новичков;
- графики;
- heatmap;
- аналитика потоков.

## 10. Критерий готовности целевого продукта

Система считается близкой к целевой, когда можно:

1. Открыть dashboard новичков.
2. Выбрать любого новичка.
3. Увидеть не только наличие данных, а статистику и оценки.
4. Открыть вкладки по этапам.
5. Посмотреть собеседование и его оценку.
6. Посмотреть учебных агентов и оценки по этапам продаж.
7. Посмотреть реальные звонки начало/середина/выпуск и динамику.
8. Посмотреть операционку.
9. Посмотреть выпускной тест.
10. Увидеть итоговую рекомендацию.
11. Открыть HTML-отчёт.
12. Сравнить новичка с другими.
