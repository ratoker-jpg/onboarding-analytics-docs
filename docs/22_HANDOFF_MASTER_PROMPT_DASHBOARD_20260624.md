### 3.2. Экспорт full bundle

Для тестового кандидата:

```bash

```

Для реального кандидата заменить `GTRAIN02` на нужный внутренний `base_key`.

Ожидаемый результат: все доступные блоки выгружены, команда не падает на secret-scan, файл создан в `tmp/`.

### 3.3. Запуск master prompt

Использовать промпт:

```text
prompts/codex/full_candidate_card_analysis_v1.md
```

На вход дать:

```text
tmp/GTRAIN02_full_bundle.json
```

На выходе нужен строго JSON по контракту:

```text
full_candidate_card_v1
```

Целевой файл результата:

```bash
tmp/GTRAIN02_full_candidate_card_result.json
```

### 3.4. Dry-run import

Сначала только dry-run:

```bash
node scripts/import_full_candidate_card.js --file tmp/GTRAIN02_full_candidate_card_result.json --dry-run
```

Dry-run должен показать:

- top-level validation PASS;
- calls semantic checks PASS;
- calls count не смешан с training agents;
- реальные звонки остались только в `calls`;
- `training_agents` отдельным блоком;
- `ops` отдельным блоком;
- отсутствующие данные не выдуманы.

### 3.5. Live import

Только если dry-run чистый:

```bash
node scripts/import_full_candidate_card.js --file tmp/GTRAIN02_full_candidate_card_result.json
```

После live import перезапустить сервер:

```bash
fuser -k 8020/tcp 2>/dev/null || true
pkill -f "run_forever.sh" 2>/dev/null || true
pkill -f "node.*server" 2>/dev/null || true

nohup ./run_forever.sh > app.log 2>&1 &
sleep 3
tail -n 80 app.log
```

## 5. Правила для master prompt

Master prompt должен быть жёстким. Нельзя просить «красиво написать отчёт» без контрактов.

Критичные правила:

1. **Не смешивать реальные звонки и учебных агентов.**
   - real calls: только `calls_start`, `calls_middle`, `calls_final`;
   - training agents: отдельный блок `training_agents`;
   - training agents не влияют на `call_quality_score`.

2. **Не выдумывать отсутствующие данные.**
   - если блока нет — `missing_data: true` / `null`;
   - не заменять отсутствующую операционку догадками;
   - не подменять дату начала погружения датой тестового дня.

3. **Давать выводы управленчески, но с опорой на источники.**
   - не писать общие фразы без причины;
   - каждую зону роста привязывать к наблюдению;
   - рекомендации тренеру должны быть действиями, а не оценочными фразами.

4. **Сохранять скоринг по рубрикам.**
   - звонки: `calls_automanual_binary_v1`;
   - собеседование: `interview_binary_v1`;
   - не менять структуру вопросников внутри JSON.

5. **Финальный JSON должен проходить importer dry-run.**
   - никаких markdown-блоков вокруг JSON;
   - никаких комментариев внутри JSON;
   - все числовые поля в ожидаемых диапазонах;
   - ключи строго по schema.

## 6. Следующий продуктовый слой: нормальный дашборд новичков

После проверки full import нужно делать отдельный PR на дашборд новичков.

### 6.1. Задача дашборда

Дашборд нужен не как технический список, а как рабочий экран тренера/руководителя:

```text
Кого смотреть сейчас?
У кого риск?
У кого нет данных?
Кто готов к выпуску?
У кого просела конкретная зона?
```

### 6.2. Минимальный состав дашборда

Карточки/таблица новичков:

- ФИО;
- направление/сегмент: `Грузоперевозки · S`;
- начало погружения;
- общий скор;
- звонки;
- собеседование;
- учебные агенты;
- операционка;
- статус/риск;
- дата последнего анализа;
- кнопка «Открыть отчёт».

### 6.3. Фильтры дашборда

Минимум:

- поиск по ФИО;
- направление;
- сегмент;
- статус;
- уровень риска;
- дата начала погружения с/по;
- наличие/отсутствие анализа по звонкам;
- наличие/отсутствие full card.

### 6.4. Визуальная логика

Использовать тот же стиль, что и отчёт:

- фиолетовые акценты;
- карточки;
- статусы;
- мягкие borders;
- без технических ключей в пользовательском UI;
- читаемо для тренера, не для разработчика.

### 6.5. Не делать в первом dashboard PR

Не смешивать с:

- новым скорингом;
- изменением importer scripts;
- изменением master prompt;
- изменением схемы БД;
- массовой переработкой отчёта.

Первый dashboard PR должен быть UI/viewer-слоем поверх уже существующих данных.
