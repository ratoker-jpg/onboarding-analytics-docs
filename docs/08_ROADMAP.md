# Roadmap внедрения

## Общая логика

Не делать сразу большой dashboard.

Сначала нужно собрать чистую модель данных и карточку новичка. Dashboard без нормальных данных будет красивой, но бесполезной картинкой.

## Phase 0. Документация и фиксация логики

Статус: текущий пакет MD.

Цель:

- зафиксировать образ результата;
- зафиксировать источники;
- зафиксировать модель ключей;
- разделить GPT/Codex;
- подготовить будущие задачи.

Результат:

- ZIP с Markdown-файлами;
- единая память проекта.

## Phase 1. Нормализованная карточка новичка v1

Цель:

Создать структуру данных, из которой можно собрать отчёт по одному новичку.

Что сделать:

1. Зафиксировать поля карточки.
2. Разделить поля на:
   - автоматические;
   - ручные;
   - AI-считаемые.
3. Описать JSON-формат карточки.
4. Описать, какие источники подтягиваются по `base_key`.
5. Сделать MVP-форму / прототип формы.

На выходе:

```text
normalized_newbie_profile.json
```

Содержит:

- ФИО;
- base_key;
- собеседование;
- тестовый день;
- маршрут;
- учебные диалоги;
- реальные звонки;
- операционку;
- итоговый тест;
- AI-analysis placeholder.

## Phase 2. Серверная база и генерация ключей

Codex zone.

Цель:

Перенести ядро с Google Sheets на серверное хранилище.

Что сделать:

1. Audit-only серверного проекта.
2. Выбрать SQLite/Postgres.
3. Добавить миграции.
4. Добавить таблицы:
   - candidates;
   - stage_keys;
   - course_progress;
   - training_bot_dialogs;
   - training_bot_roles;
   - voice_bot_sessions;
   - real_calls;
   - operational_metrics;
   - final_tests;
   - report_snapshots.
5. Реализовать генерацию session_key.
6. Сохранить совместимость входа по base_key.

На выходе:

- сервер хранит кандидатов;
- server-side session_key;
- Google Sheets не primary storage.

## Phase 3. Импорт источников

Цель:

Подтянуть старые источники в серверную модель.

Источники:

- Веб-версия МВП;
- Маршрут новичка;
- Бот учебки;
- Вопросы для автомануала;
- Кроссы подбор;
- Статистика новичка;
- итоговые тесты;
- ручные звонки.

Способ:

- импорт из Google Sheets;
- ручная загрузка файлов;
- webhook/JSON/CSV от голосового бота;
- fallback ручной ввод.

На выходе:

- данные по одному base_key можно собрать автоматически частично;
- ручные поля вводятся через форму.

## Phase 4. AI-анализ карточки

Цель:

AI анализирует нормализованную карточку и возвращает структурированный результат.

Что сделать:

- промпт анализа новичка;
- промпт анализа звонка по мануалу;
- промпт анализа учебного бота;
- схема JSON-ответа;
- сохранение report_snapshots.

На выходе:

- hard_score;
- soft_score;
- learning_score;
- discipline_score;
- risk_score;
- final_decision;
- рекомендации тренеру;
- HTML/Markdown-отчёт.

## Phase 5. MVP Dashboard

Цель:

Сделать первую управленческую страницу.

Минимум:

- список новичков;
- поиск по ФИО/base_key;
- фильтры: сегмент, статус, риск;
- карточка новичка;
- scores;
- кнопка “сформировать отчёт”.

Без сложных графиков на старте.

## Phase 6. Сравнение новичков

Цель:

Выбирать нескольких новичков и сравнивать.

Что добавить:

- выбор 2–5 новичков;
- bar chart;
- heatmap;
- radar chart;
- AI-сравнение;
- вывод “кого выпускать / кому нужна практика”.

## Phase 7. Аналитика этапов и курсов

Цель:

Оценивать качество обучения, а не только людей.

Добавить:

- результаты по курсам;
- частые ошибки;
- слабые вопросы;
- сложные роли бота;
- корреляция этапа с выпуском;
- рекомендации по улучшению курса.

## Phase 8. Экспорт и отчётность

Добавить:

- HTML-отчёт;
- Markdown-отчёт;
- Excel/XLSX;
- JSON;
- Google Sheets export.

Важно:

```text
Экспорт — это выход из системы, не источник истины.
```

## Приоритеты

### Must have

- base_key/session_key;
- карточка новичка;
- ручная форма ввода;
- импорт тестового дня;
- импорт бота учебки;
- AI-анализ;
- сохранение результата.

### Should have

- реальные звонки старт/середина/финал;
- операционка;
- итоговый тест;
- базовый dashboard;
- HTML-отчёт.

### Could have

- сравнение новичков;
- графики;
- сегментная аналитика;
- аналитика курсов;
- автоматический webhook.

### Later

- права доступа;
- роли пользователей;
- SLA/уведомления;
- интеграции с внешними сервисами;
- полностью автоматическая загрузка звонков.

# Обновление v2: новый порядок после Phase 2B

После реализации серверного сбора данных стало понятно, что нельзя сразу делать “красивый dashboard”.

Правильный порядок:

## Phase 3A — Analytics Model & Rubrics

Статус: следующий шаг.

Цель:

- зафиксировать шкалы оценки;
- зафиксировать критерии по этапам;
- зафиксировать итоговые scores;
- зафиксировать правила статусов и рисков;
- спроектировать tables/endpoints для аналитики.

Результат:

- единая методология оценки;
- понятно, что считать и как показывать.

## Phase 3B — Manual Analytics UI

Цель:

- дать тренеру возможность вручную выставлять оценки по шкалам;
- сохранить candidate_scores;
- показать read-only dashboard на ручных оценках.

## Phase 3C — AI Analysis

Цель:

- AI оценивает собеседование, звонки, учебных агентов и итоговый профиль по утверждённым шкалам.

## Phase 3D — HTML Report Snapshots

Цель:

- формировать read-only HTML-отчёт;
- хранить историю отчётов.

## Phase 3E — Full Dashboard

Цель:

- фильтры;
- сравнение новичков;
- графики;
- heatmap;
- аналитика потоков.
