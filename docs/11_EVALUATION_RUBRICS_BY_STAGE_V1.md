# EVALUATION_RUBRICS_BY_STAGE_V1.md

Дата фиксации: 23.06.2026  
Проект: `onboarding_cross_node`  
Назначение: базовая методология оценки новичков по этапам обучения, чтобы dashboard, карточка новичка, AI-анализ и HTML-отчёт считали одно и то же.

---

## 1. Главный принцип

Система должна показывать не только:

```text
данные есть / данных нет
```

а отвечать на вопросы:

```text
как новичок справился?
где просадка?
какой риск?
можно ли выпускать?
что контролировать после выпуска?
```

`Completeness` — это техническая готовность данных.  
`Analytics / Scores` — это смысловая оценка новичка.

---

## 2. Единая шкала 0–100

Все оценки приводятся к шкале `0–100`.

| Балл | Уровень | Интерпретация |
|---:|---|---|
| 0–20 | критический провал | Навык почти отсутствует, выпуск невозможен без доработки |
| 21–40 | слабый уровень | Много ошибок, нужен плотный контроль и повторная отработка |
| 41–60 | базово, нестабильно | Понимание есть, но качество плавает, выпуск только с контролем |
| 61–75 | рабочий минимум | Можно выпускать при понятном плане контроля |
| 76–89 | сильный уровень | Новичок уверенно справляется, риски точечные |
| 90–100 | отличный уровень | Сильный запас, можно использовать как пример |

---

## 3. Итоговые статусы

| Статус | Когда ставим |
|---|---|
| `готов к выпуску` | Нет критичных рисков, основные оценки выше рабочего минимума |
| `готов с контролем` | Есть отдельные зоны риска, но они управляемые |
| `нужна доработка` | Есть существенные просадки, выпуск сейчас рискован |
| `не рекомендован` | Критичные hard/soft/discipline риски |
| `недостаточно данных` | Нельзя сделать честный вывод, данных мало |

---

## 4. Risk score

`risk_score` — отдельная шкала 0–100, где 100 = максимальный риск.

| Risk score | Уровень |
|---:|---|
| 0–25 | низкий |
| 26–55 | средний |
| 56–75 | высокий |
| 76–100 | критичный |

### Красные флаги, которые повышают риск

- опасные обещания клиенту;
- системное непонимание продукта/условий;
- неумение выявлять потребность;
- отсутствие динамики после обратной связи;
- операционная недисциплина;
- просрочки клиентских целей;
- плохие/пустые комментарии;
- низкая трубка;
- провал выпускного теста по критичным темам;
- сильные soft-риски на собеседовании;
- отказ принимать обратную связь.

## 5. Этап 1 — Собеседование

### Источники

- TXT-транскрибация собеседования;
- заметки HR;
- заметки тимлида;
- вопросы:
  - `Вопросы на мотивацию`;
  - `Вопросы на мотивацию 2.0`;
  - `Вопросы на антихрупкость`;
  - `Собесы`.

### Что оцениваем

#### 5.1. Soft interview score

| Критерий | Что смотрим |
|---|---|
| Мотивация | Зачем человеку роль, насколько мотивация реалистична |
| Зрелость | Как объясняет опыт, решения, ошибки |
| Структурность | Отвечает ли по сути, умеет ли собирать мысль |
| Коммуникация | Тон, ясность, контактность |
| Рефлексия | Понимает ли свои зоны роста |
| Антихрупкость | Как реагирует на сложные вопросы/давление |
| Обучаемость | Готов ли менять поведение по обратной связи |

### Шкала soft interview

| Балл | Описание |
|---:|---|
| 0–20 | избегает ответов, нет зрелости, коммуникация разваливается |
| 21–40 | ответы поверхностные, слабая рефлексия, мотивация неубедительная |
| 41–60 | базово отвечает, но нестабильно и без глубины |
| 61–75 | нормальный рабочий уровень, можно обучать |
| 76–89 | сильная коммуникация и зрелость |
| 90–100 | очень сильный профиль, высокая обучаемость и самостоятельность |

#### 5.2. Hard / sales fit interview score

| Критерий | Что смотрим |
|---|---|
| Понимание продаж | Есть ли логика этапов продажи |
| Выявление потребности | Не прыгает ли сразу в презентацию |
| Презентация | Умеет ли связать продукт с задачей клиента |
| Возражения | Не спорит, не давит, уточняет причину |
| Закрытие | Договаривается о следующем шаге |
| Клиентский контекст | Использует ли вводные клиента |
| Корректность | Не обещает лишнего |

#### 5.3. Ролевка

По ролевке обязательно фиксировать:

```text
цель ролевки
сценарий клиента
что продавал / проверял
как начал контакт
какие вопросы задал
как презентовал
как обработал возражения
как закрыл
какая была обратная связь
принял ли обратную связь
```

### Вывод по собеседованию

Должны сохраняться:

- `soft_interview_score`;
- `sales_fit_score`;
- `motivation_score`;
- `learning_potential_score`;
- `roleplay_score`;
- `interview_risk_level`;
- `summary`;
- `strengths`;
- `growth_zones`;
- `red_flags`;
- `recommendation_for_training`.

## 6. Этап 2 — Тестовый день

### Источники

- Web MVP;
- лист `Кандидаты`;
- лист `Сборка`;
- голосовой бот выгрузка;
- тесты;
- открытые ответы;
- питчи;
- рефлексия.

### Что выводим

- дата первого входа;
- прошёл / не прошёл блоки;
- результаты тестов;
- открытые ответы;
- оценка голосового агента;
- питч до обратной связи;
- питч после обратной связи;
- рефлексия;
- общий вывод.

### Что оцениваем

| Оценка | Что значит |
|---|---|
| `test_theory_score` | Насколько понимает теорию, продукты, процессы |
| `test_practice_score` | Как применяет знания в задачах/агентах |
| `feedback_adoption_score` | Улучшился ли после обратной связи |
| `product_understanding_score` | Понимает ли продуктовые условия |
| `baseline_hard_risk` | Базовый hard-риск на входе |

### Шкала тестового дня

| Балл | Описание |
|---:|---|
| 0–20 | не понимает базу, не справляется с заданиями |
| 21–40 | много ошибок, логика продукта/продажи слабая |
| 41–60 | частично справляется, нужны доработки |
| 61–75 | рабочий минимум для входа в обучение |
| 76–89 | сильный старт |
| 90–100 | очень сильная база и быстрая адаптация |

## 8. Этап 4 — Учебные агенты / тренажёрные диалоги

В UI использовать название:

```text
Учебные агенты
```

Технический источник может называться `bot_training`.

### Источники

- `Бот учебки / Траскрибация`;
- `Бот учебки / Роли`;
- `Бот учебки / Результаты`;
- source link с точным external key.

### Что выводим

- external key;
- количество диалогов;
- даты;
- роли клиентов;
- сценарии;
- результат по каждому диалогу;
- ошибки по этапам продаж;
- динамика.

### Оценка одного диалога

| Блок | Вес | Что смотрим |
|---|---:|---|
| Контакт | 15% | поздоровался, представился, причина звонка, удобно ли говорить |
| Выявление | 25% | понял бизнес, задавал вопросы, не прыгал в презентацию |
| Презентация | 20% | связал продукт с потребностью, говорил конкретно |
| Возражения | 15% | уточнил причину, не спорил, дал аргумент |
| Закрытие | 15% | согласовал действие/следующий шаг |
| Корректность | 10% | не обещал лишнего, соблюдал условия продукта |

### Шкала учебных агентов

| Балл | Описание |
|---:|---|
| 0–20 | диалог неуправляемый, критичные ошибки |
| 21–40 | есть контакт, но продажа почти не строится |
| 41–60 | базовые этапы есть, но много пропусков |
| 61–75 | рабочий минимум |
| 76–89 | уверенный диалог |
| 90–100 | сильная продажа, можно использовать как пример |

## 9. Этап 5 — Реальные звонки

### Блоки

- `calls_start` — звонки начала;
- `calls_middle` — звонки середины;
- `calls_final` — звонки выпуска.

### Важное решение

Поле `продукт` не должно быть обязательным и не должно определять оценку.

Заменить на:

```text
Контекст / тема звонка
```

Примеры:

- ОБ;
- ЭДО;
- фонды;
- общая кросс-продажа;
- неизвестно;
- другое.

Оценка идёт по этапам продаж и мануалу, а не по названию продукта.

### Оценка звонка

| Блок | Вес | Что смотрим |
|---|---:|---|
| Установление контакта | 15% | удобно ли говорить, представился, причина звонка |
| Выявление потребности | 25% | вопросы про бизнес, задачи, проблемы, текущий процесс |
| Презентация | 20% | продукт связан с потребностью клиента |
| Возражения | 15% | не спорит, уточняет, отвечает по сути |
| Закрытие | 15% | следующий шаг, подключение, встреча, договорённость |
| Сегментная адаптация | 5% | учитывает сегмент клиента |
| Корректность обещаний | 5% | нет запрещённых обещаний |

### Оценка динамики

Сравнить:

```text
начало → середина → выпуск
```

Выводить:

- что улучшилось;
- что не изменилось;
- какие ошибки повторяются;
- можно ли выпускать без плотного контроля.

### Итоговые поля

- `calls_start_score`;
- `calls_middle_score`;
- `calls_final_score`;
- `call_growth_score`;
- `call_quality_score`;
- `call_risk_flags`;
- `call_recommendation`.

## 12. Итоговые шкалы новичка

### Основные итоговые scores

| Score | Из чего собирается |
|---|---|
| `hard_score` | тестовый день, учебные агенты, реальные звонки, выпускной тест |
| `soft_score` | собеседование, учебные агенты, реальные звонки |
| `learning_score` | погружение, принятие ОС, динамика звонков, рефлексия |
| `discipline_score` | погружение, операционка, сроки, статусы |
| `call_quality_score` | учебные агенты + реальные звонки |
| `ops_score` | трубка, XSALES, просрочки, статусы, комментарии |
| `final_test_score` | выпускной тест |
| `risk_score` | красные флаги + слабые scores |
| `overall_score` | агрегированный итог |

### Предварительная формула overall_score

```text
overall_score =
  hard_score * 0.25 +
  soft_score * 0.15 +
  learning_score * 0.15 +
  discipline_score * 0.15 +
  call_quality_score * 0.20 +
  final_test_score * 0.10
```

`risk_score` не входит напрямую в average. Он используется как ограничитель статуса.

### Ограничители статуса

- если `risk_score >= 76` → нельзя `готов к выпуску`;
- если `hard_score < 50` → минимум `нужна доработка`;
- если `discipline_score < 50` → минимум `готов с контролем`, часто `нужна доработка`;
- если есть опасные обещания клиенту → минимум `готов с контролем`, при системности `нужна доработка`;
- если нет данных по реальным звонкам → статус не выше `готов с контролем`;
- если нет собеседования и нет звонков → `недостаточно данных`.

## 15. Следующий правильный шаг

Не писать новый dashboard сразу.

Сначала:

```text
Phase 3A — зафиксировать модель аналитики и шкалы
```

Потом:

```text
Phase 3B — ручной слой оценок и read-only dashboard
```

Потом:

```text
Phase 3C — AI-анализ по этим шкалам
```

Потом:

```text
Phase 3D — HTML report snapshots
```
